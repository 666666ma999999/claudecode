# /rescue - Codexタスク引き渡しコマンド

行き詰まった時や、Codexの完遂力・深い思考が必要な場面でタスクをCodexに引き渡す。

## 用途

- Claude Codeで3回修正失敗（3-Fix Limit到達）時
- 複雑な実装でCodexの努力量が必要な場面
- 根本原因の特定が難航している時
- 非エンジニアタスク（資料レビュー、文章校正等）のCodex委譲

## 引数

```
/rescue [説明テキスト or ファイルパス]
```

| 引数 | 説明 |
|------|------|
| (なし) | 対話的に状況をヒアリングしてからCodexに送信 |
| テキスト | そのままCodexに転送 |
| ファイルパス | ファイル内容を読み取ってCodexに転送 |

## 実行手順

### 1. コンテキスト収集

以下の情報を自動収集する:

- **task.md**: 存在すればタスク目的・成功基準・進捗
- **git diff**: 未コミットの変更差分
- **git log -5**: 直近5コミットのメッセージ
- **エラーログ**: 直近のテスト失敗やサーバーエラー（あれば）

### 2. 構造化テンプレートの組み立て

```
## Goal
{達成したいこと — task.mdの目的 or ユーザー入力}

## Current State
{現在の状態 — git diff要約, 実装済み部分}

## What Was Tried
{試したこと — 直近の変更履歴, 修正試行}

## Failure Signal
{失敗の兆候 — エラーメッセージ, テスト失敗, 期待と実際の乖離}

## Constraints
{制約 — プロジェクトルール, 技術制約, 時間制約}

## Wanted Output
{期待する出力 — 具体的な成果物, 形式, 品質基準}
```

### 3. Codexに送信

`mcp__codex__codex` で構造化テンプレートを送信する。

Codexからの追加質問や確認が必要な場合は `mcp__codex__codex-reply` で継続する。

### 4. 結果の統合

Codexからの回答を受け取り:
1. 提案された解決策をユーザーに提示
2. ユーザーの承認を得てから適用
3. 適用後はバッチ検証ループに従って検証

## 注意事項

- Codexへの送信内容に**機密情報（APIキー、パスワード等）を含めない**
- Codexの回答は鵜呑みにせず、適用前に妥当性を確認する
- 大量のコード（1000行超）を送信する場合は要約してから送信
- rescueの結果で実装変更した場合、通常のバッチ検証・implementation-checklistフローに従う
