# /adversarial-review — 敵対的クロスレビュー統括

あなたはレビュー統括役です。以下のフローを順守し、各承認ゲートでは必ず停止してユーザーの判断を待ってください。

## 議題

$ARGUMENTS

（議題が空なら、直前の会話で議論中のテーマを 1 行で要約し「この議題で進めてよいか」を先に確認する）

## 大原則（全フェーズ共通）

- **レビュアー構成の正直な明記**: レビュアー A = このセッションと同じモデルの**別コンテキスト**（general-purpose サブエージェント・Fable 系）/ レビュアー B = **異モデル**（`mcp__codex__codex`・model 指定はしない＝環境既定。特定モデル名の指定が効くかは未検証）。報告書では「同モデル別文脈」と「異モデル」を区別して表記し、「2 つの異なる AI が一致」と過大表示しない
- **アンカリング防止**: 両レビュアーには**ユーザーの議題原文だけ**を渡す。統括役の要約・仮説・補足を混ぜない。2 体は互いの出力を見せず独立実行（フェーズ 1）
- **軽量モード**（引数に `--light`）: フェーズ 2 を省略し、フェーズ 1 の統合とゲート①のみで完結（小さい議題用・所要 1/3）
- **棲み分け**: Plan モード中の実装計画・設計リスクのレビューは既存の `plan-adversarial-review` スキルを使う。本コマンドは**企画・方針・悩みの議論**用

## フェーズ 1: 一次レビュー（2 体並行・独立）

1. レビュアー A（サブエージェント）とレビュアー B（Codex・sandbox=read-only）を**並行**で呼び出し、それぞれに独立で実行させる:
   - 問題点・抜けている前提・失敗しそうな箇所を**重要度順**に列挙
   - 確信が持てない指摘には冒頭に「**推測:**」を付ける
2. 統括役が両者の出力を統合する:
   - 重複する指摘はマージし「**両者一致**」とマーク
   - 片方のみの指摘は出典（A=同モデル別文脈 / B=異モデル）を併記
   - 全体を重要度順に再ソートして提示
3. **【承認ゲート①】** 3 択を取る（指摘が多くても選択肢はこの 3 つに固定）:
   - 「全指摘を前提に進む」/「番号を指定して除外する（その他欄に番号）」/「議題や前提を修正する」
   - 承認があるまでフェーズ 2 に進まない
   - ⚠️ **提示方法（focus mode 対策・mistakes.md askuserquestion-after-text）**: 統合指摘一覧を書いた**同じメッセージで AskUserQuestion を呼ばない**（ツール UI が本文を差し替え、一覧がユーザーの画面から消える。2026-07-14 実害）。**一覧＋「返事は一言で（全部前提/番号除外/前提修正）」の地の文だけでターンを終える**か、一覧の要点を AskUserQuestion の question/options 自体に収容する

## フェーズ 2: 敵対的クロスレビュー（--light では省略）

1. レビュアー A に B の指摘一覧を、B に A の指摘一覧を渡し、**相手の指摘だけ**を敵対的にレビューさせる
   - 敵対的レビューの定義: **擁護禁止**。誤り・過剰な悲観/楽観・根拠不足・見落としの指摘に徹する。自分自身のフェーズ 1 指摘の弁護はさせない
2. 報告フォーマット（固定・4 部構成）:
   1. **攻撃を生き残った指摘**（= 両者とも崩せなかった結論）
   2. **崩れた/弱まった指摘**（誰がどう崩したか）
   3. **新たに割れた論点**（各レビュアーの主張と根拠を併記）
   4. **統括役の裁定** — 「Fable 裁定」と明示する（統括役はレビュアー A と同モデルのため構造的に A 寄りになり得る旨を自覚した表記）。推測には「推測:」。**割れたまま残った論点はユーザー裁定へ回す**（勝手に決めない）

## フェーズ 3: 実装計画（実装を伴う議題のみ）

- 手順の分解（ステップごとに成果物を明記）
- 繰り返し使える部分の Skill / コマンド切り出し候補
- 並行実行できる部分のサブエージェント分割案（依存関係も示す）
- **【承認ゲート②】** 計画を提示し、ユーザーの OK が出るまで実装に着手しない（提示方法はゲート①の ⚠️ と同じ — 計画本文と同一メッセージで AskUserQuestion を呼ばない）

## 完了時

- 結果はユーザーが読む形（vault reports/ 等・vault-report-writing 準拠）で残すか、チャット完結でよいかを議題の重さで判断（迷ったら聞く）
- 有効だった論点・確定した方針は既存の記録先（decisions.md / mistakes.md / playbook）へ `/save` で振り分け（新ファイルを作らない）
