# /adversarial-review - Codex敵対的レビューコマンド

コード変更に対して、Codexの深い思考力で**批判的・敵対的視点**のレビューを実行する。
通常の `/review` と異なり、前提条件・設計判断・ロジックの妥当性に疑問を投げかける。

## opponent-review との違い

| | /adversarial-review | opponent-review |
|--|----|----|
| 実行エンジン | Codex (mcp__codex__codex) | SubAgent 2体 |
| 速度 | 高速（Codex単体） | 重い（並列SubAgent） |
| 対象 | コード変更・差分 | 設計判断・アーキテクチャ |
| 用途 | 実装の批判的検証 | トレードオフ・代替案の探索 |

## 引数

```
/adversarial-review [オプション]
```

| オプション | 説明 |
|-----------|------|
| (なし) | git diff HEAD で全変更を敵対的レビュー |
| --staged | ステージング済みの変更のみ |
| --file PATH | 特定ファイルのみ |
| --last-commit | 直前のコミットを対象 |

## 実行手順

### 1. 変更内容の取得

`/review` と同じ方法で git diff を取得する。

### 2. コンテキスト収集

- task.md（存在する場合）→ 目的・成功基準・前提条件
- CLAUDE.md → プロジェクト制約
- 変更ファイルの周辺コード（変更行の前後50行）

### 3. Codexに敵対的レビュー依頼

`mcp__codex__codex` を使用。以下のプロンプトテンプレートで依頼:

## 検証観点（全て実行すること）

### 1. 前提への疑問
- この変更の前提条件は本当に正しいか？
- 暗黙の仮定は何か？その仮定が崩れるケースは？

### 2. ロジックの穴
- エッジケース・境界値で破綻しないか？
- 並行処理・タイミング問題はないか？
- エラーパスで状態が不整合にならないか？

### 3. 見落としリスク
- この変更が影響する範囲で、テストされていない箇所は？
- パフォーマンス劣化の可能性は？
- 将来の変更を困難にする設計上の罠は？

### 4. 代替案
- もっとシンプルなアプローチはないか？
- この変更は本当に必要か？既存の仕組みで解決できないか？
