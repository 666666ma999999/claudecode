概要
うまくいっている点: あなたは、プライマリセッションをメモリ観測エージェントが記録する、独創的なデュアルエージェントワークフローを構築しました。これは、ほとんどのユーザーが考えていない先進的なインフラです。実装前にアーキテクチャ優先の分析（変数マッピング、状態機械監査、段階的計画）を行い、その後Codexレビューを実行するという規律は、一貫して最良の成果を生み出しています。各修正後に8ステップCMSパイプラインをエンドツーエンドでストレステストする体系的なアプローチは、優れたエンジニアリングの厳密さを示しています。 優れた成果 →
課題: Claudeの側では、実際の根本原因を見つける前に、誤った仮説を複数回追いかけるパターンが続いています。例えば、字幕が実際には画像であるのに、テキストベースの字幕検出をデバッグするのに時間を費やしました。あなたの側では、多くのセッションが堅実な計画で終わるものの、コードが一切適用されず、以前のセッションの修正が持続しないことが頻繁にあります。これは、終了前の検証ゲートがないためです。アーキテクチャコンテキスト（CMSが実際にどのようにレンダリングするかなど）を前もって提供すれば、Claudeの無駄な探索を大幅に削減できます。 問題が発生するケース →
すぐに試せる改善策: メモリ観測ワークフローは最も頻繁な活動ですが、ほとんどのセッションは部分的にしか成功していません。ヘッドレスモードを使用して、観測エージェントをスクリプトから非対話的に実行すれば、より信頼性が高く手間がかからなくなります。「STEP 1-8パイプライン全体を実行して失敗を報告」や「このセッションからメモリ観測を作成」のような繰り返しワークフローにカスタムスラッシュコマンドを設定すれば、セットアップ時間の無駄がなくなります。また、フックを使用してコミット後にエンドツーエンド検証テストを自動実行すれば、修正が実際に適用されない問題を捕捉できます。 試すべき機能 →
将来的な可能性: モデルがより高性能になるにつれて、8ステップパイプラインは自律的な自己修復の完璧な候補になります。単一コマンドを起動するだけで、すべてのステップを実行し、ログとDOM状態を検査して失敗を診断し、修正を適用し、グリーンになるまで再実行するシステムを想像してください。メモリ観測の実践も大きなアップグレードの準備ができています。手動の部分的なキャプチャではなく、継続的エージェントがコードベースの生きた知識グラフを構築し、新しいセッションの開始時に関連するコンテキスト（既知のバグ、変数マッピング、レンダリングの癖）を自動的に提示し、今日最も時間がかかっている再発見サイクルを排除できます。 今後の展望 →

作業内容

8ステップCMS登録ワークフロー自動化 約20セッション
日本のCMS（Rohan）向けの占い・PPVコンテンツを対象とした、自動化された8ステップ製品登録パイプラインの開発とデバッグ。Claude Codeは、エンドツーエンドテスト、各ステップでの失敗診断（STEP 2サマリーバグ、STEP 8メニュー反映問題、セッション永続化問題）、フロントエンドとバックエンドファイル全体での修正実装、堅牢性改善の反復に大いに活用されました。孤立メニューアイテムフィルタリング、ブラウザ自動化の失敗、黒丸サマリーバグ、CMS自動送信の拒否などの問題に多大なデバッグ労力が費やされました。
PPV字幕・本番環境検証チェック 約10セッション
PPVコンテンツの本番環境検証チェックシステム（State Machine 2）の構築とテスト。字幕検出、ページレンダリング検証、Playwrightブラウザ自動化を使用した決済フロー確認が含まれます。Claude Codeは、19項目チェックフローアーキテクチャの設計、段階的検証ステージの実装、字幕検出失敗のデバッグ（字幕が画像でテキストではないことを発見）、型不整合と競合状態の修正、本番ページに対する自動ブラウザベーステストの実行に使用されました。
コードベースアーキテクチャ分析・ドキュメント化 約8セッション
Rohanプロジェクトのアーキテクチャの包括的な調査とドキュメント化。フロントエンド/バックエンド間の50以上の変数のマッピング、デュアル状態機械ワークフローの分析、APIエンドポイントとデータフローのドキュメント化、FE/BE統合改善の計画が含まれます。Claude CodeのRead、Grep、Bashツールは、登録ワークフローと確認ワークフロー間の変数依存関係の追跡、変数マッピングドキュメントの作成、クロスシステム統合の詳細な実装計画の作成に広く使用されました。
コード品質・パフォーマンス・堅牢性改善 約7セッション
プロジェクト全体のテスト堅牢性、パフォーマンス最適化、コード品質を対象とした多段階改善計画の実装。Claude Codeは、ブラウザイベントハンドラのメモリリーク修正、Codexレビューに基づく入力検証の追加、STEP 8で92%の速度改善の達成、フィールド名不一致の修正、ポーリングセーフなUI更新の実装、追跡されたPlaywrightアーティファクトとサーバーログによる20MBリクエスト制限エラーを解決するための.gitignore更新を含むgit操作の管理に使用されました。
メモリ観測エージェント操作 約5セッション
プライマリコーディングセッションを監視・記録する副次的Claude-Mem観測エージェントの実行。ワークフロー発見、バグ修正、アーキテクチャ発見の構造化XML観測を作成。Claude Codeは、8ステップワークフローシステム、変数マッピング、字幕修正、診断セッションに関する観測をキャプチャし、検索可能なメモリドキュメントに整理するために使用されましたが、観測はセッション終了前に部分的にしかキャプチャされないことが多かったです。

Claude Code の使い方

あなたは複雑な多段階自動化ワークフローを実行するパワーユーザーです。具体的には、日本のPPV製品システム向けの8ステップCMS登録パイプラインを運用しており、Claude Codeをマラソンセッションを通じてデバッグパートナーおよび実装エンジンとして使用しています。6週間にわたる282セッション、合計約790時間という非常に高いエンゲージメント頻度を維持しています。最も特徴的な行動は、プライマリコーディングセッションを監視・記録する並列「メモリ観測」エージェントを実行することです。これが、最大の目標が`memory_observation_recording`（22セッション）である理由です。このメタレイヤーアプローチは、アドホックなバグ修正ではなく、組織的知識を体系的に構築していることを示しています。あなたはClaudeに広範囲に実行させる傾向があります。Bash（3,149回）、Read（2,475回）、Grep（1,212回）の多用は、介入前にClaudeにコードベースを深く調査させていることを示しています。しかし、Claudeが軌道を外れた場合は割り込んで方向転換させます。これは、Claudeが単純なyes/no質問に答えずにセッション全体を調査に費やしたり、手動修正を反映するように自動テストが更新されていないことを繰り返し指摘しなければならなかった摩擦イベントから明らかです。

あなたのインタラクションスタイルは特異性をエスカレートさせる反復デバッグと最もよく表現されます。通常、広範なリクエスト（「STEP 1-8フルテストを実行」）から始め、失敗が表面化するとそれを掘り下げます。STEP 2のタイミング問題、STEP 8の孤立メニューフィルタリング、字幕の画像対テキスト検出などです。詳細な事前仕様を提供することはめったになく、実行失敗を通じて要件を発見します。これは、Claudeが字幕がテキストではなく画像であることを発見する前に複数のデバッグ反復を経た字幕チェックの経緯で明らかです。また、実装前に計画を要求するパターンも示しています。いくつかのセッションで包括的な段階ベースの計画（SM2の6段階計画、堅牢性改善計画など）が作成され、その後のセッションで実行されました。摩擦プロファイル（21の誤ったアプローチ、16のバグコード、8の誤解リクエスト）は、探索的なミスを許容しながらも、Claudeが意図を誤解したり、以前に議論した修正を適用しなかった場合には強く押し戻すことを確認しています。摩擦にもかかわらず、99の「概ね満足」評価は、特にワークフローを定義する深いデバッグ（18の成功）と複数ファイル変更（9の成功）に関して、全体的なパートナーシップを非常に生産的と感じていることを示しています。

特徴的なパターン: 並列メモリ観測エージェントを使用した長期にわたる複数セッション自動化キャンペーンを編成し、Claudeに広範囲に探索させながら、焦点を失ったり以前のコンテキストを組み込まない場合には断固として介入します。

優れた成果

6週間にわたる282セッションで、複雑な複数ステップCMS自動化パイプラインを構築し、体系的なデバッグとアーキテクチャドキュメント化へのすばらしい献身を反復的に洗練させました。

メモリ観測エージェントアーキテクチャ
専用のClaude-Mem観測エージェントがプライマリコーディングセッションをリアルタイムで監視・記録し、ワークフロー、変数マッピング、バグ修正計画の構造化XML観測を作成する、洗練されたデュアルエージェントワークフローを開発しました。セッション間で組織的知識を保存するこのメタ認知アプローチは、非常に先進的であり、一時的なコーディングセッション上に永続的な知識レイヤーを構築していることを示しています。
体系的な8ステップパイプラインデバッグ
完全な8ステップCMS登録パイプラインを体系的に構築・ストレステストし、エンドツーエンドテストを繰り返し実行し、各失敗ポイントを掘り下げました。STEP 2のタイミング問題からSTEP 8の孤立メニューフィルタリングバグ、ブラウザ自動化拒否まで。各修正後にフルパイプラインを再実行する粘り強さと、Claudeの修正が実際に適用されなかったことを特定する能力は、強力なエンジニアリング規律と深いシステム理解を示しています。
アーキテクチャ優先計画とCodexレビュー
実装前にClaudeを包括的な分析に向けて一貫して誘導しています。50以上の変数にわたる完全な変数マッピングドキュメント、デュアル状態機械アーキテクチャ監査、コードを書く前の段階的実装計画を要求しています。その後、実装された変更に対してCodexレビューサイクルを実施し、問題を早期に捕捉します。この規律ある計画-実装-レビューループは、最高の満足度と最も成功した成果をもたらしました。

問題が発生するケース

セッションは、根本原因の繰り返し誤診、実装前に停止する計画、セッション間で持続しない問題の再発に大きく影響されています。

誤ったデバッグアプローチによる無駄な反復
Claudeが実際の根本原因を発見する前に、誤った仮説を複数回追いかける状況に頻繁に遭遇し、行き止まりでかなりの時間を浪費します。システムアーキテクチャに関する事前コンテキスト（字幕が画像であってテキストではないなど）を提供したり、反復前に仮定を検証するようClaudeに依頼することで、デバッグサイクルを大幅に節約できます。
  • 字幕チェックは、Claudeが最終的に字幕が検索可能なテキストではなく画像としてレンダリングされていることを発見するまで、ログ追加とタイムアウト調整の複数デバッグ反復を経て失敗しました。これは事前に1回の検査で確立できた構造的事実です。
  • STEP 2は、実際の根本原因がsite_id URLの問題だったのに、タイミング問題と誤診されて2回失敗し、実際には問題ではなかった待機状態とタイムアウトのデバッグ労力を浪費しました。
計画段階でセッションが終了し実装なし
多くのセッションが、詳細な計画や調査で終わりますが、実際のコード変更は適用されず、既にスコープされた作業を実行するためだけにフォローアップセッションを開始する必要があります。オープンエンドな分析リクエストではなく、より小さな実装中心のプロンプトにタスクを分割して、各セッション内で動作するコードを確実に取得することを検討してください。
  • 機能が修正されたかどうかを尋ねたところ、Claudeはセッション全体の15分を広範なコードベース探索と計画に費やし、直接答えたりコード変更を行ったりすることはなく、実用的な価値はゼロでした。
  • 20MBリクエストサイズ修正、黒丸サマリーバグ、CMSリフレクトボタン機能、STEP 8字幕登録はすべて「計画作成済みだが実装未開始」で終わりました。少なくとも6セッションで修正ではなく計画を受け取りました。
セッション間で修正が持続しない
以前のセッションからの修正が実際に適用されていない、または再び壊れる状況に繰り返し遭遇し、同じ問題の再調査を余儀なくされます。これは検証のギャップを示唆しています。各修正後にClaudeにエンドツーエンド検証テストを実行させ、セッションを終了する前に自動テストコードが手動修正を反映していることを明示的に確認することを検討してください。
  • STEP 2サマリー黒丸とSTEP 8リフレクトボタンの以前の修正が実際に機能していないことを2回報告する必要があり、完了したはずの作業の再調査と修正が必要でした。
  • Claudeは、あなたが複数回明示的に指摘するまで、Phase 1-4,6の手動テスト修正がcheck_playwright.pyに反映されていないことに気づかず、テストコードと本番コードが静かに乖離していました。

試すべき機能

CLAUDE.md への追加提案

これをClaude Codeにコピーして、CLAUDE.mdに追加してください。

複数のセッションで、ユーザーが直接的な回答を求めているのにClaudeが広範なコードベース調査を行い、あるセッションは単純な質問に答えなかったために「未達成」と評価されました。
摩擦データは、ユーザーが「以前の修正が実際に機能していない」（STEP 2サマリーとSTEP 8リフレクトボタン）と複数回報告し、再調査が必要だったことを示しています。
複数のセッションで、ユーザーが「複数回明示的に指摘」するまでClaudeが手動修正後に自動テストを更新せず、繰り返し摩擦が発生しました。
セッションデータは、macOS上でLinux形式のコマンドが使用されて失敗したことを示しており、これは繰り返す必要のない簡単な環境事実です。
少なくとも4セッションが追跡アーティファクトとログからの20MB以上のリクエストサイズエラーの影響を受け、複数のセッションがこの再発問題の診断のみに費やされました。
複数のセッションで、ClaudeがSTEP 2をURLの問題であるのにタイミング問題として誤診し、STEP 8バグがフィルタリングロジックを中心に再発し、大幅なデバッグ時間を浪費しました。
字幕がテキストではなく画像であることを発見するために複数のデバッグ反復が浪費されました。これは今後の後戻りを防ぐためにドキュメント化すべき事実です。

これをClaude Codeにコピーすれば、セットアップしてくれます。

カスタムスキル
繰り返しワークフロー用の単一/コマンドで実行できる再利用可能なプロンプト。
あなたへの提案理由: STEP 1-8パイプラインテストを多くのセッションで頻繁に実行しており、繰り返しのメモリ観測ワークフローがあります。両方とも毎回再説明する代わりに単一コマンドスキルにできます。
mkdir -p .claude/skills/pipeline-test && cat > .claude/skills/pipeline-test/SKILL.md << 'EOF' # Full Pipeline Test 1. Ensure server is running and .gitignore excludes large files 2. Run STEP 1-8 sequentially, logging each step's result 3. If STEP 2 fails, check site_id URL first (not timing) 4. If STEP 8 fails, check prefix filtering and residual items 5. Report summary table: step, status, duration, errors 6. Update automated test code if any manual fixes were applied EOF
フック
特定のライフサイクルイベントで自動実行されるシェルコマンド。
あなたへの提案理由: 最大の摩擦は「誤ったアプローチ」（21件）と「バグのあるコード」（16件）です。コミット前フックでリンター/型チェッカーを実行し、編集後フックで大きなファイルをチェックすれば、多くの問題を自動的に捕捉できます。
# .claude/settings.jsonに追加： { "hooks": { "preCommit": [ { "command": "find . -name '*.log' -size +5M -not -path './.git/*' | head -5 | xargs -I{} echo 'WARNING: Large file detected: {}'" } ], "postEdit": [ { "command": "python -m py_compile $CLAUDE_EDITED_FILE 2>&1 || true" } ] } }

メモリ観測ワークフローを統合
第1位の活動（22セッション）はメモリ観測ですが、ほとんどが「一部達成」です。標準化して自動化してください。
メモリ観測エージェントパターンは分析されたセッションの44%を消費していますが、観測が完了前にセッションが終了するか、観測エージェントがプライマリセッションからのコンテキストを欠いているため、一貫して部分的な結果しか達成できません。現在の形式では、これは高労力・低収穫パターンです。並列観測エージェントを実行する代わりに、各作業セッションの終わりに構造化形式で観測を書くか、ヘッドレスモードを使用して事後的にログをバッチ処理することを検討してください。
Claude Code に貼り付け:
このセッションの終了時、終了前に、.claude/observations/session_$(date +%Y%m%d).mdに構造化観測ファイルを作成してください。セクション：## Changes Made, ## Bugs Found, ## Variables Discovered, ## Open Issues, ## Next Steps。検索性のためにXMLタグを使用してください。

今後の展望

282セッションにわたる790時間は、CMS自動化と複数ステップワークフローテストを中心とした洗練されたAI支援開発プラクティスを明らかにしています。これは、対話型デバッグから完全自律パイプラインオーケストレーションへと飛躍する準備ができています。

自律的自己修復テストパイプラインオーケストレーション
STEP 1-8登録ワークフローは、最も頻繁で摩擦の多い作業負荷です。誤ったアプローチ（21）とバグコード（16）が最大の摩擦源であり、ほとんどが1つのステップの出力が次を壊すカスケード失敗に起因します。自律エージェントは、完全な8ステップパイプラインを継続的に実行し、任意のステップでの失敗を検出し、ログとDOM状態を検査して根本原因を診断し、修正を適用し、再実行できます。すべて人間の介入なしで。単一コマンドを起動して戻ってくると、すべての8ステップがグリーンで、何が自動修正されたかの変更履歴があることを想像してください。
始め方: Claude CodeのTaskツール（既に5番目に多用されており496回の呼び出し）を使用して、並列サブエージェントを生成します。ステップごとに1つの隔離診断用と、シーケンス、リトライロジック、修正適用を管理するオーケストレーターエージェントを組み合わせます。既存のPlaywrightブラウザ自動化と組み合わせてください。
Claude Code に貼り付け:
あなたはRohan CMS 8ステップ登録ワークフロー用の自律テストパイプラインオーケストレーターです。ゼロ人間介入でSTEP 1-8をエンドツーエンドで実行することがあなたの仕事です。 各ステップについて： 1. 既存の自動化コードを介してステップを実行 2. 失敗した場合、サブエージェント（Taskを使用）を生成して、ログを読み、ブラウザ自動化を介してDOMを検査し、関連するバックエンドコードを分析して根本原因を診断 3. コードベースに直接修正を適用 4. 失敗したステップを再実行して検証 5. 次のステップに進む 注意すべき既知の失敗パターン： - STEP 2: サマリー黒丸（●）バグ、AJAX待機タイミング、site_id URL問題 - STEP 8: 孤立メニューアイテムフィルタリング、リフレクトボタンクリック失敗、プレフィックスベースフィルタリングバグ、600sタイムアウト - ステップ間: セッションデータ永続化失敗、インメモリキャッシュの陳腐化 すべての8ステップを完了した後、構造化レポートを生成：ステップごとのタイミング、適用された修正（diffあり）、成功率、残存問題。いずれかのステップが修正試行後に3回失敗した場合、診断をログに記録し、次のステップに進んでください。 現在のテストランナーコードと最新のserver.logを読んでから、実行を開始してください。
並列エージェントコードベース分析と修正適用
既にTask/TaskUpdateパターン（合計878回の呼び出し）とメモリ観測エージェントを使用していますが、現在これらは単一問題に対してシリアルに実行されます。並列サブエージェントを使用すれば、クロスカッティングバグが現れたときに、フロントエンドフォームマッピング、バックエンドAPIエンドポイント、ブラウザ自動化レイヤーを同時に調査できます。データはmulti_file_changes（9の成功）とFE/BE統合分析にまたがるセッションを示しています。これらは、スタック全体で並列に作業するエージェントによって3倍速くなる可能性があり、各エージェントは自分のレイヤーに焦点を当てた独自のコンテキストウィンドウを持ちます。
始め方: 複数のClaude Code Taskサブエージェントを同時に起動します。フロントエンドHTML/JS分析用に1つ、バックエンドPython調査用に1つ、テストコード更新用に1つ。各自がTaskUpdate経由で発見を報告し、オーケストレーターを使用してすべてのレイヤーにわたって調整された修正を統合・適用します。
Claude Code に貼り付け:
Rohanプロジェクトのクロスカッティングバグの調整された複数エージェント調査が必要です。Taskを使用して3つの並列サブエージェントを生成してください： Agent 1 - フロントエンド調査員: - 影響を受けるワークフローステップに関連するすべてのHTMLテンプレートとJavaScriptファイルを読む - すべてのフォームフィールド、AJAXエンドポイント呼び出し、DOM操作をマップ - レポート: フィールド名、期待されるデータ型、イベントハンドラー、不整合 Agent 2 - バックエンド調査員: - 同じワークフローステップのすべてのPythonルートハンドラー、データモデル、ユーティリティ関数を読む - すべてのAPIエンドポイント、期待されるパラメータ、データベース操作、レスポンス形式をマップ - レポート: パラメータ名、型期待、検証ロジック、不一致 Agent 3 - テスト・統合調査員: - このステップのすべてのPlaywrightテストコードとブラウザ自動化スクリプトを読む - すべてのセレクタ、待機条件、アサーション、既知の回避策をマップ - レポート: 脆弱なセレクタ、ハードコードされたタイムアウト、欠落しているアサーション、実際のFE/BE動作とのギャップ 3つのエージェントすべてが報告した後、発見を統合して： 1. 変数マッピングテーブル（FEフィールド名 → APIパラメータ → BE変数 → DBカラム） 2. 特定された不一致または型不整合 3. 正確なファイルパスと行番号を含む優先順位付き修正計画 4. すべての修正を実装し、関連するテストを実行し、グリーンならコミット。
継続的メモリ駆動開発ループと自動ドキュメント化
第1位の目標カテゴリはmemory_observation_recording（22セッション）です。プライマリセッションで何が起こるかを観測エージェントがドキュメント化する新しいプラクティスを構築しました。しかし、今日これは手動で、部分的（ほとんどの観測セッションで「一部達成」）で、実際の開発ループから切り離されています。自律システムは、すべてのセッションを継続的に観測し、コードベースアーキテクチャ、既知のバグ、修正パターン、変数マッピングの生きた知識グラフを構築し、各新しいセッションの開始時に関連するコンテキストを積極的に提示できます。字幕が画像でテキストではないこと、または同じ孤立メニューフィルタリングバグを再診断する必要はもうありません。
始め方: Claude CodeのTodoWriteツール（223回の呼び出しは既に作業項目を追跡していることを示します）を構造化CLAUDE.mdメモリファイルと組み合わせて、自己更新知識ベースを作成します。バックグラウンドエージェントに、既存のXML観測形式を使用して、ワークフローステップとコンポーネントでインデックス化された観測を継続的に書かせます。
Claude Code に貼り付け:
あなたはRohanプロジェクトの継続的メモリとドキュメンテーションエージェントです。責任： 1. 知識ベース監査: プロジェクト内のすべての既存のメモリファイル、スキルドキュメント、観測XMLを読む。ドキュメント化されているものと欠落しているものの構造化インデックスを作成。 2. バグパターンデータベース: コードベースのgit履歴と既存のメモから、遭遇したすべてのバグの検索可能なカタログを構築、整理基準： - ワークフローステップ（STEP 1-8、SM2チェック） - コンポーネント（フロントエンド/バックエンド/ブラウザ自動化） - 根本原因カテゴリ（タイミング、型不一致、セレクタ脆弱性、データ永続化、フィルタリングロジック） - 適用された修正と検証ステータス 3. 変数マッピング更新: 現在のコードベース状態をスキャンしてvariable_mapping.mdを更新 — すべてのFE→API→BE→DBマッピングがまだ正確であることを検証。最後の更新以降の新しい変数または変更されたフィールド名にフラグを立てる。 4. セッションコンテキストジェネレーター: 知識ベースに基づいて、新しいセッションの開始時にロードすべきCLAUDE.mdコンテキストブロックを生成。含めるべき内容： - 最も頻繁な失敗ポイント上位5つとその実証済み修正 - 各ワークフローステップの現在の状態（最後のテスト結果、既知の問題） - 重要なアーキテクチャ事実（例：「字幕は画像としてレンダリングされ、テキストではない」、「fixedCode001アイテムは特別なプレフィックス処理が必要」） - アクティブなTODOアイテムとその優先順位 5. すべての出力を/memoryディレクトリに構造化ファイルとして書き込み、一貫した命名と相互参照を使用して、将来のエージェントがgrepで関連するコンテキストを即座に見つけられるようにする。 プロジェクト構造とすべての既存ドキュメントを読んでから、ステップ1-4を順番に実行してください。
