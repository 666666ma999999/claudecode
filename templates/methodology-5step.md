# 作業メソドロジー（0層 + 6手順 + メタ層）— 新プロジェクト デフォルト

> prime_suite で確立した「データ駆動で施策を回す」作業の型。
> 新プロジェクト着手時にこれを置き、各ステップの **問い** に自分のプロジェクトのデータ・ツールで答えて、下のタスクを埋める。
> ※ タスクは汎用の出発点。**転用するのは各ステップの「問い」**。自分のプロジェクト用に具体タスクを書き換えて使う。
> 全体像: `0. 基盤 →（① → ② → ③ → ④ → ⑤ → ⑥ →①へ戻る）← メタ層が全体を守る`

## 0. 基盤・環境構築（着手前に一度）
**問い: 分析・作業を再現可能に回せる土台ができているか？**
- [ ] 実行環境を固定する（Docker / 依存バージョン固定で再現性）
- [ ] データ・成果物の保管と引き継ぎを決める（どこに置く / 別PCでも続けられるか）
- [ ] 機密・PII のガードを敷く（pre-commit / マスク / commit 禁止リスト）

## ① 既存データの分析
**問い: このプロジェクトが既に持つ実測データは何か？ 記憶や勘でなく、それを起点にしたか？**
- [ ] 既存の実データを洗い出す（過去ログ / 実績 / 既存分析）
- [ ] 主要KPIのベースラインを出す（観測窓を揃える・全期間混在の過大評価に注意）
- [ ] そのデータから現状を把握する（推測で始めない）

## ② 実運用確認
**問い: このプロジェクトの「最新の実態」は何か？ 古い集計だけで判断していないか？**
- [ ] 「最新の実態」にあたるものを定義する（取得パイプラインがあれば最新を取る）
- [ ] 過去集計だけで決めず、最新の実態と突合する
- [ ] 古いデータと最新のズレを記録する

## ③ 公式・一次ソース確認
**問い: 前提にしている仕様・数値・ベストプラクティスを、記憶でなく一次ソースで確認したか？**
- [ ] 仕様/数値を公式・一次ソースで確認する
- [ ] 出典URLと確認日を残す（90日以内・古ければ再確認）
- [ ] 未確認のものは「未確認」と明示する

## ④ 施策・タスク起案
**問い: 案は固定フォーマットか？ 効果を盛っていないか？ やめる条件を先に決めたか？**
- [ ] 固定フォーマットで起案（何を / なぜ / 期待効果 / 使うデータ / 詳細）
- [ ] 効果見積りは楽観補正する（盛らない）
- [ ] 撤退基準（やめる条件・閾値）を事前に決める

## ⑤ ファクトチェック・反証
**問い: 投入前に「その前提は本当か」を壊しにいったか？ 証拠で裏取りしたか？**
- [ ] 主要な前提を壊しにいく（反証実験）
- [ ] grep / 実データで主張を裏取りする（記憶でなく証拠）
- [ ] 結果を①の実データに戻す（次サイクルへ）

## ⑥ 可視化
**問い: 分析・結論を人がレビューできる形にしたか？**
- [ ] 結果を図表化する（数字の羅列でなく図で）
- [ ] 構造・関係を Canvas 等で可視化する
- [ ] レビュー用に1枚にまとめる（1-pager / 司令塔ノート）

## メタ層. 手順を「仕組み」で守らせる（全体を包む）
**問い: ①〜⑥が放置・飛ばし・思い込みで崩れないようガードを作ったか？**
- [ ] 飛ばしやすい必須チェックを hook で物理ゲート化する
- [ ] 逸脱しやすい手順を「規約」として明文化する
- [ ] pre-commit / テストで機械検証を回す
- [ ] 確定知見と不確実性ラベルを台帳化し、放置・劣化を防ぐ
