# プロンプト分類 v2 — 工程(P0-P5)×機能(Bサブタイプ11種)の二次元分類

## なぜ二次元か
v1(A-H)は「機能」の一次元分類だった。実測5,516件の再分析で判明した v1 の粗さの正体は2点: ①最大クラスB(起案・発注)が1クラスに潰れており、実際は性質の異なる11サブタイプ1,724件(goal 69 / require 88 / collect 53 / analysis 92 / ideate 177 / implement 360 / dispatch 417 / verify 97 / rule 96 / report 69 / env 206)を含んでいた。②「いつ打つか」の工程軸が無く、同じ ideate でも P2(施策言語化)と P5(定常運用)で前後の遷移とスロットが違う。プロジェクトを「作る」には両軸が必要。

## 軸1: ライフサイクル工程 P0-P5(行動プロキシの週次指紋で機械分離可能)
- **P0 種火**(1-4日): プロジェクト未存在。単発依頼 or 素朴なビジネス質問。例: prime_suite 2026-04-30「google広告の管理画面をみているのですが、一般的にどの部分に着目して運用するべきですか？」
- **P1 現実接続・データ搬入**(1-2週): AIが触れない管理画面・実データを人間が運搬。指紋=仲介率(「これですか？」「よく分かりません」)19-20%(安定期0-5%)
- **P2 設計・施策言語化**(3日-1週): plan.md 文法の確立。施策固定フォーマット+敵対検証。prime_suite 5/7 の1日で完成
- **P3 建設・集中改修**(1-3週): 実装バースト(週457件ピーク)→launchd 自動運転で件数1/10に急減
- **P4 検証・運転監視**(1-3週): 長時間ラン babysitting。指紋=監視率28-29%+定時checkブロック11-12%(pokeca 4/27-5/4週)
- **P5 運用・裁定**(無期限): 起案→敵対検証→実行→ファクトチェック→記録のPDCA。派生テーマが P0' へ分岐(pokeca 5/30 裁定調査・6/14 SP500・6/26 下落で3回反復)

## 軸2: 機能(v1 A-H との対応)
- **v1-B 起案(約17%)** → 11サブタイプへ分解(上記)。40字以上の実質起案937件が v1 の17%とほぼ整合。dispatch の約8割・implement の約半分は40字未満の最小形(「やって」「実行」「pushして」)で、v1 では裁定/是正に誤算入されていた可能性が高い
- **v1-A 是正(窓により12-21%)** → v2 では CORRECT(「違いますよ」「全然ダメ。」)+HELP(「よくわかりません」226件)として **B の外の品質ゲート層** に置く。生成コーパスの負例。是正の直後に rule(「失敗例として保存して」)が続く3連鎖が再発防止の型
- **v1 裁定(約24%)** → RULING(「a」「1」「これ採用で。」a単独123回)。v2 では独立クラスではなく **Bサブタイプ間の遷移接続点**(ideate→verify→**RULING**→dispatch)として保持。短文裁定は全期間11-41%で常時最大成分=ユーザーの主業務
- その他(進捗質問「続いてますか？」40回・「終わりましたか？」29回)→ QUESTION。全Bサブタイプは QUESTION にサンドイッチされる(状況確認→B発注→結果確認)

## 二次元マトリクス(工程ごとの主力サブタイプ・実測)
| 工程 | 主力B | 周辺機能 | 代表原文 |
|---|---|---|---|
| P0 | goal / collect(可否確認) | QUESTION(素朴質問) | 4/30「現状の管理画面から状況をDLできますか？」 |
| P1 | collect / env(秘密ゲート) / rule(台帳化) | QUESTION「これですか？」/ HELP | 5/1「githubにpushしますが、機密データを…pushされない構成になっていますか？」 |
| P2 | require / ideate / verify / report(施策表) | RULING / CORRECT(様式矯正) | 5/7「施策ベースで答えて欲しいです。何の施策を立てるから、この資料が必要です」 |
| P3 | implement / dispatch / env(worktree) | RULING(着手承認) | 5/20「影響するファイル、データの流れ、テストの観点を整理してから実装に入って」 |
| P4 | dispatch(最小形) / verify(定時check) | QUESTION(生存確認) / RULING | 5/2「[build-data 4h check…]…300字以内で報告」 |
| P5 | ideate / verify(三点セット) / rule(/save) / report(valutMD) / goal(再起案) | CORRECT(辛辣化) / RULING | 6/27「agent teamで考えて、codexで敵対的レビューして、必ずファクトチェックはすること」 |

## 遷移文法(創造の背骨・両projectで3回以上反復実証)
**ideate → verify → RULING → dispatch** が中核連鎖。プロジェクト創造の7拍子(pokeca 5/30以降3回反復): ①自然文起案(数値条件埋込)→②agent team+codex 独立クロスチェック→③worktree分離実装→④Codex GO/NO-GOレビュー→⑤実画面/実データで人間検品→⑥ルール化(「メモしてください」)→⑦launchd/CIで運用固定。二層プロンプト体制(ユーザー短文トリガー→AIが構造化委譲文を清書→別窓実行)が汚染12-18%の正体であり、これ自体が分身の設計図。

## 2. プロジェクト創造プレイブック（14 手・実録裏付きのみ）

## プロダクト3類型の genesis の型(原文実証)

- genesis = テーマ提示→「codex , agent teamで考えて」起案→敵対的レビュー→plan→実装、の**儀式をフルで踏む**。plan.md/phase 管理を最初から持ち、ゴールが KPI(CPA・利回り等)で定義される。

### 類型3: 環境(~/.claude)— 「痛点駆動型・genesis 不在」

- 3類型は独立でなく**教訓の供給網**: 小型の失敗(art_013 破損・0imp惨敗・レポート様式不満)→環境の rule/skill 化(auto-commit hook・vault-report-writing)、大型の痛点(広告知識の忘却・pokeca の _INBOX)→環境の recall hook・命名規約。環境は他2類型の genesis の「排熱」を制度に変換する装置。

### 2. 工程の違い

- **事業**: ①実物要求→②目視不満→③詰まったら設計外注→④人間ゲート宣言→⑤永続化最小装置→⑥完走後に仕組み昇格→⑦ゴール自問・縮小。**実物先行・制度後置**。genesis 週で骨格が決まり、定常運用(make_article)か縮小凍結(autopost)へ収束する。
- **環境**: 失敗体験→違和感の一言→ルール化委託→敵対レビュー→**機械強制の強度階段**(ルールに書く<警告hook<pendingファイル機械連結<denyで物理遮断)→1行スモーク→効果確認→**再発時は恒久対策へ格上げ**。**失敗先行・成果物=制度そのもの**。「完成」がなく Phase 0〜7(4/27〜7/2)の永続改善。独立ロードマップは存在せず、常に「事業作業中に環境の穴を踏む→その場で環境セッションを開く」往復運動(recall hook は prime_ad の議論中に、命名規約は pokeca への _INBOX 展開中に誕生)。
- **固有規範**: **2-strike 規範** — 1回目=その場修正、2回目=制度化要求(「これ先日、恒久的な対策をしませんでしたか？」6/22・「以前それは推奨されなかった」5/24)。のちに MEMORY.md「恒久対策を好む」として本人が明文化。

### 10.4 分身がプロジェクト新設時に行うデータ設計手順
1. 1. project type判定（rules/42）: ①実コードor実データを持つ→Type A（repo連携）②plan+施策+分析+進捗の2種以上を継続運用→Type B（vault-only）③それ以外→Type C（単発ノート1枚）。Cならファイル1枚で終了・構造を作らない。
2. 2. Type Aの初期配置は最小2点だけ: vault `02_Ai/<project>/<project>_ope.md`（MOC 1ファイル・frontmatter 6フィールド: project/type/folder/categories/last_updated/tags）と repo側 `plan.md`+`tasks/`。vault側にprogress/strategy/measures等の実体ファイルは作らない（住所+サマリーをMOCに持つ）。
3. 3. `wiki/meta/project-registry.md` にentry追記（SessionStart hookの注入対象・全group同一registry）。ここに載せないとAIの自動想起ループから外れる。
4. 4. プロンプト受け口は `prompts/<project>_INBOX.md` 1枚（上🔵やってほしいこと/下📒記録）。spot/・done/・_README・execution-logは作らない。定期実行が必要になった時だけ `prompts/scheduled/`+launchd。
5. 5. 横断器は再利用宣言: 決定→decisions.md、教訓→mistakes.md、Xネタ→x-article-stock.md、X記事→make_article。project専用のdecisions/wiki/_indexを新設しない（K-3廃止済・N-4新固有名はdecisions起票必須）。
6. 6. 命名を最初に固定: 全ファイル`<project>`または`<scope>-`prefix・vault全体unique・汎用名単体禁止・ローマ字(ASCII)。bareにできるのは外部ツールが名前を固定する物（CLAUDE.md/README/plan.md/tasks/NOW.md/phase-tracker.md/lessons.md）だけ。判定は「basenameが単独表示されてprojectが分かるか」1問。
7. 7. データの器: repo `data/raw/`(append-only・gitignore)・`data/processed/`(再生成可・gitignore)・取得元台帳`docs/data-sources.md`・系譜`docs/data_lineage.yaml`。secretsはどこにも置かない(${VAR}参照)。vault側の外部取り込みは`.raw/<topic>/<topic>-YYYY-MM-DD.<ext>`。
8. 8. 各器の更新様式を作成時に宣言: append-only（履歴・証跡・raw）/de-dup統合（教訓）/固定名上書き（毎日見るボード・導出ビュー）。machine-generatedな上書きファイルはgitignore検討（2-PC sync衝突源）。
9. 9. ユーザー可視性の線引きを確認: このユーザーが日常的にrepoを開くか？開かないなら「見て判断する物（優先順位・決定・状態ボード）」はvault正本にし、repo側は機械用ミラーと明示。MOC同期義務（repo更新と同セッションでlast_updated当日化）を守る。
10. 10. 自動化はまだ作らない: hook/skill/レポート自動生成はN=1では見送り、手動運用2-3回の実績後に昇格判断。作る時は hook本体+routing/CLAUDE.md+SKILL.md の3層同時更新＋注入/起動経路（SessionStart/UserPromptSubmit/PreCompact）が実在するか確認。
11. 11. 増築の判定を習慣化: 新情報が来るたび「既存ファイルのH2セクション追加で収まるか」を先に問い、収まるなら新ファイルを作らない。субprojectディレクトリは複数mdが必要になってから切る（空階層禁止）。運用知が溜まってきたら初めて `<project>-playbook.md`（SessionStart注入）や`<project>-impl-notes.md`を追加。
12. 12. 環境雛形は既存コマンドで敷く: `/init-project`（環境基盤）+`/methodology`（作業の型）。リビング参照実装は `02_Ai/AI_adscrm/`（MOC構造・prefix命名・reports/prompts集約の完成形）。

### 10.5 未言語化だった判断（今回初めて言葉になったもの・11 項）
- 「司令塔」の判定基準が未言語化: ユーザーの言う司令塔は type=moc でも plan でもなく「開いた瞬間に判断が完結する、毎日見る1枚」。どのファイルが司令塔かはルール上のtypeでなくユーザーの日常動線で決まる（2026-06-15にAIがMOC=司令塔と誤認して露呈）。『人間の視線が週1以上通るか』という暗黙の判定軸はどのrulesにも書かれていない。
