# 応答構造（focus mode対応）詳細 — 現場Claudeの提案が読めない問題の解消

> **薄い索引は `CLAUDE.md`「## 応答構造（focus mode対応）」**。本ファイルは背景・実例・テンプレ全文・hook契約の SSoT。
> 機械保証の実体 = `hooks/stop-evidence-footer.sh`（5-Tier）。一次資料 = `state/readability-work/`（gitignore）。

## 1. なぜ作ったか（背景）

**問題**: 実装作業中のClaudeが返す「提案」チャット応答が、根拠・出典・注意書き・強制フッターと同一平面に混ざり、
肝心の**結論/決定ポイントが読めない**。

**一次実例**（セッション `a9bf41de` 2026-07-04）:
1. **L145** Claude が提案（画像OCR採用可否のA/B/C案）を返す。だが冒頭は「ファクト判定」で、
   ユーザーが答えるべき問い（A/B/Cどれ?）は〔事実判定→トークン計算式→3反対理由→出典〕の**壁の最下部**。
2. **L149** Stop hook（旧 `stop-evidence-footer.sh`）が「末尾に🔍根拠フッターを付けよ」とブロック。
3. **L153** Claude が**フッター1行だけの新メッセージ**を出力。
4. focus mode は「最終テキストメッセージだけ」を表示 → 画面には**フッターしか出ず本文（提案）が消失**。
5. **L166** ユーザー「**すいません、どこに回答が書いてあるのですか？**」← これが指摘の実物。
6. **L171** Claude が全文再掲。決定ブロックに「**← これが『回答＝あなたへの質問』です**」と注記する羽目に＝構造失敗の自認。

**2連鎖の症状**: ①決定ポイントの埋没（構造）②根拠フッター×focus modeの回答消失（フック挙動）。

設計＝エージェントチーム（多視点解析→3哲学設計→判定→統合）。敵対レビュー＝Codex（判定：差し戻し・6修正）。
設計仕様 = `state/readability-work/design-proposal.md`。

## 2. 応答テンプレ（現場Claudeの骨格・1メッセージ完結）

提案/判断を返すときは、この順序で **1つのメッセージ**にまとめる（focus mode 対策＝最終メッセージ単体で完結）。

````markdown
## ✅ 結論 / 決めること
**推奨: A案** — <推奨を1文。ユーザーが取るアクション/選ぶ選択肢を書く（事実判定ではない）>

決めてください（1つ選ぶ）:
- **A案（推奨）** — <一言>
- **B案** — <一言>
- **C案** — <一言>

---

### 根拠 / 但し書き
- <結論を支える要点だけ。ファクト判定（「半分本当」等）はここへ降格>
- <条件付きの但し書き。数値は要点のみ>

<details><summary>詳細（計算式・実測値・長い引用）</summary>

<精密な数値・出典引用・長文をここに隔離>
</details>

---
🔍根拠: ファクト[実確認] 出典[<file:line or URL>] 現運用[参照済/対象外] 二重記載[チェック済]
Sources:
- [<title>](<url>)
````

**固定規則**:
1. **決定ブロックが最上部**（推奨1文＋選択肢2つ以上）。「結論」＝ユーザーが取るアクション/選ぶ選択肢。
2. `---` で区切り、根拠を中段（長文は `<details>`）。
3. `---` の下に 🔍根拠フッターと Sources を隔離。
4. **フッター/根拠だけの最終メッセージ禁止**（focus mode で回答消失になる）。

## 3. 機械保証：`hooks/stop-evidence-footer.sh`（5-Tier契約）

soft（本規約）＝良い初回応答を出させる主レバー。machine（hook）＝壊れた最終メッセージを止める安全網。役割分担。

| Tier | 発火条件 | ブロック内容 | 対象 |
|---|---|---|---|
| **Tier0 hook再開・回答消失** | `stop_hook_active=true` ∧ 同ターン直前の assistant テキスト≥250字(compact) ∧ 最終<100字 ∧ 直前の40%未満 ∧ 結論先頭でない | 「hook対応の一言で停止せず、回答本文を丸ごと再掲して1メッセージで」 | 2026-07-10 rohan 実害（dup-guard誤検知→1行継続で回答消失）の恒久対策。テスト=`hooks/tests/test-stop-evidence-footer-t0.sh` |
| **Tier1 回答消失** | フッター有り ∧ 実本文が実質空 ∧ 結論先頭でない | 「フッターだけ。冒頭に結論ブロックを置き本文＋フッターを1メッセージで」 | 最優先・トリガー語/短文除外より前 |
| **Tier2 co-location** | `is_choice_proposal` ∧ フッター無し | 「決定ブロック先頭・根拠中段・フッター末尾を1メッセージで」 | L145初回形 |
| **Tier3 埋没是正** | `is_choice_proposal` ∧ フッター有り ∧ 結論が先頭に無い | 「決定ブロックを先頭へ移せ」 | **推奨つき選択依頼限定**＝完了/検証報告・純粋質問は巻き込まない |
| **Tier4 原ガード** | 断定 ∧ ¬`is_choice_proposal` ∧ フッター無し ∧ ≥200字 | 従来の🔍根拠フッター要求（#1/#3/#5 の act-time 強制） | 完了/検証報告 |

**`is_choice_proposal` ＝ `has_propose` ∧ `offers_choice`**（＝Claude が**推奨を述べ、かつ選択を求めている**応答のみ restructure 対象）:
- `offers_choice`: ユーザーに選択/決定を**明示的に**求めている（`選んでください`/`決めてください`/`どれにしますか`/`どちらにしますか` 等）。bare「A案」言及だけでは発火**しない**（過去形「A案を採用した」等の完了報告を巻き込まない）。
- `has_propose`: 推奨/提案/すべき/おすすめ 等の推奨語。**推奨のない純粋な選択質問**（例「AとBどちらにしますか、選んでください」）は `is_choice_proposal=False` で **restructure 対象外**（推奨ゼロの場面に「推奨1文」を強制しない＝stance捏造回避・validator新規2）。
- **既知ギャップ（許容）**: 「推奨＋選択肢を出すが `選んでください` 等の依頼語を書かない」埋没提案は machine 非捕捉 → **soft規約（CLAUDE.md）に委譲**（過剰検知を避けるトレードオフ・validator新規1）。
- **`decision_at_top`**: 「最上部3行に決定見出し/推奨マーカー ∧ 実体(選択肢2つ以上 or 推奨文)が先頭10行内に co-located」（先頭に空の `## 結論` を貼り実体を下方に埋めるデコイは不合格）。

**ループ安全**: `stop_hook_active` を全免除フラグに**使わない**。理由＝当環境は複数 Stop hook が併存し
`stop_hook_active` は「どの Stop hook か」を区別しない共有フラグ。全免除に使うとフック誘発の footer-only 再ターン（L153）を
素通しして事故を再現する（Codex指摘）。代わりに `state/evidence-footer/<session_id>.txt` に
`message-hash:tier` を記録し、**同一メッセージは1回だけ再ブロック**（前例＝`stop-dup-guard.sh` の session_id 状態）。
セッション上限（6）で最終遮断。

**誤検知回避（温存）**: **推奨のない純粋な質問**（`is_choice_proposal=False` なので質問tail除外が効く）・短文（<200字）・完了/検証報告への結論構造強制なし・fail-open。
※「埋没した推奨つき選択依頼」（`is_choice_proposal` ∧ 結論が先頭に無い）だけは質問tail除外の対象外＝restructure させる（実L145形の取りこぼし防止）。

**未検証（正直に）**: `stop_hook_active` の正確な意味論・`last_assistant_message` フィールドの有無は当環境で未確証。
本設計は前者に依存せず（hash状態でループ安全）、後者を使わず（transcript逆走＝ローカル実績あり）に成立する。Tier3 は緩和可能なノブ。

**golden 検証**: `state/readability-work/golden_test.py`（全14ケース PASS）。
基本: c1 footer-only→block / c2 同一hash2回目→pass / c3 提案footer無→block / c4 埋没→block / c5 完全体→pass / c6 完了報告footer→pass /
r1 短文→pass / r2 質問tail→pass / r3 Tier4→block。
敵対形状(validator硬化): **c7 実L145形(質問tail締め・埋没)→block** / **c8 完了報告が過去形でA案言及+footer→pass** / **c9 デコイ空##結論+実体を下方に埋没→block** /
**c10 推奨+選択肢だが選択依頼語なし→pass(soft委譲・既知ギャップ)** / **r4 推奨のない純粋質問→pass(誤ブロックしない)**。

## 4. スコープ棲み分け（重複ドリフト防止）

- **チャット応答の構造** ＝ 本規約 + `stop-evidence-footer.sh`（＝この文書）。
- **vault の人向け成果物（レポート/1-pager/ダッシュボード）** ＝ `vault-report-writing` skill（BLUF・二層化はそちらの担当）。
  `docs/bunshin-*` の「plain-language-bluf」メモはレポート系の思想であり、チャット応答一般の規約は本文書が正本。
- 既存 `CLAUDE.md`「エラー報告・バグ修正（症状→原因→選択肢2つ以上）」＝本規約と補完関係。選択肢文化は保ち、配置（先頭）を足すだけ。

## 5. 却下した重い選択肢（過剰設計回避）

- 新規 Stop hook 増設 → 末尾積みの土壌を悪化。既存1本の外科改修で足りる。
- `permissions.deny` で構造強制 → deny は禁止語/コマンド用。整形には不適。
- `output-style` で全ターンに骨格強制 → 軽い事実確認・雑談まで儀式化。将来レバーに温存。
- UserPromptSubmit 先頭ゲート（毎ターン「結論照合」宣言） → 毎ターンの重い儀式。再発が数字で続けば後追い増設。
