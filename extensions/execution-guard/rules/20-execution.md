# 実行ガード

## ブロッカープロトコル

以下発生時、即座に停止してユーザーに確認:
- プランの指示が不明確
- テスト/検証が失敗
- プラン外の変更が必要と判明
- 想定外の依存関係を発見

禁止: ブロッカーを推測で回避。

## バッチ実行方式

タスクを3つずつバッチ化、検証ポイントを設ける。
- 単純変更: 最大5/バッチ
- 高リスク変更: 1/バッチ

## プラン作成基準

含める要素: Goal, Architecture, Tech Stack, Tasks, Verification。
タスク粒度: 2-5分、原則1ファイル、1論理変更。
各タスクに推奨: ファイルパス, 関数名, コード, 検証コマンド。
