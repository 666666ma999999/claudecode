# 実装品質ルール

## エラー報告・バグ修正

**報告フォーマット:**
1. 症状: エラーメッセージの正確な引用（Sentry URL該当時のみ）
2. 原因: コード追跡→根本原因（ファイル名:行番号）
3. 経緯: 過去の変更履歴・スキルとの不整合
4. 選択肢: 複数の修正方針（最低2つ）+ トレードオフ

**自律バグ修正:**
- バグレポートを受けたら、手取り足取り教えてもらわず自力で修正する
- ログ・エラー・失敗テストを確認し、自分で根本原因を特定・解決する
- ユーザーのコンテキスト切り替えをゼロにする
- 失敗しているCIテストは言われなくても修正する

## 実装完了チェック（自動発火）

Edit/Writeでコード変更後、ユーザーへの完了報告前に `implementation-checklist` スキルを実行。
除外: ドキュメントのみ、設定ファイルのみ、1行以下の軽微修正。

完了前の自問: 「スタッフエンジニアはこれを承認するか？」

## 自己改善

**メモリシステム:**
- Claude-Mem: 活動記録（自動）
- Memory MCP: 知識蓄積（意図的保存 — 「覚えておいて」、新方針決定時、再利用知見発見時）

**学習ループ:**
- ユーザーから修正を受けたら必ず プロジェクトの `tasks/lessons.md` にパターンを記録する
- 同じミスを繰り返さないよう、自分へのルールを書く
- ミス率が下がるまでルールを徹底的に改善し続ける
- セッション開始時に、プロジェクト関連のlessonsをレビューする
