# 状態機械テスト生成

「pending/running/completed/failed ＋自由形式JSON」で始めた状態管理は、承認・依存・再試行が入った瞬間に壊れる。実装前に**遷移表を書き、表からテストを機械的に導出**する（出典: reading-factory Phase C 敵対レビュー 2026-07-14）。

## 手順

### 1. 遷移表を先に書く（コードより先）
- [ ] 状態の全列挙（人間待ち状態は専用に: `awaiting_approval` 等）
- [ ] イベント（API呼出・完了通知・タイムアウト・人間操作・再起動）× 現状態 → 次状態 の表
- [ ] **表に無い遷移は全部拒否**が仕様（暗黙の「まあ動く」を残さない）

### 2. 表から導出するテスト（各行が1ケース）
- [ ] 全**許可遷移**が成功する
- [ ] 全**禁止遷移**が拒否される（任意状態→completed の直接更新が通る実装は欠陥）
- [ ] **CAS**: 遷移は「現状態を条件に UPDATE」。並行して同じ遷移を2回送って**1回だけ**成功する
- [ ] **二重送信**: 同一イベントの重複配送で副作用が1回（冪等性キー）
- [ ] **再起動**: 各状態でプロセスを落として復元。外部処理が進行中の状態は「failed に落とす」でなく**再照会で回収**（有償APIなら二重課金の点検 → paid-generation-gate-audit）
- [ ] **reaper/タイムアウト干渉**: 長寿命状態（人間承認待ち）が reaper・起動時 sweep に殺されない
- [ ] **版失効**: 上流入力（種・パラメータ）の版が変わったとき、下流の承認・成果物が自動失効する

### 3. 実装上の縛り
- [ ] 状態名・イベント名・payload 形状は**1箇所の契約定義**を正本に（複数人/複数エージェントで並行実装する前に凍結）
- [ ] 自由形式JSON列に工程情報を足し続けない（工程が3つを超えたら workflow/step/decision の別エンティティ化を検討）
- [ ] ジョブ粒度と同時実行制限（per-key 1件・global N件）の衝突を先に洗う
