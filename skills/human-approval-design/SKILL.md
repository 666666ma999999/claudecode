# 人間承認（Human Approval）設計

「人の判断が必要」を安易にボタン1個で実装すると、**承認が証拠にならない・承認待ちがタイムアウトで死ぬ・古い承認が生き残る**の3事故が起きる（出典: reading-factory Phase C 敵対レビュー 2026-07-14）。

## チェックリスト

### 1. 承認の証拠性（何を承認したか）
- [ ] 承認対象は **artifact の content hash に束縛**（「ジョブID承認」では中身の差し替えに気づけない）
- [ ] 承認時に**表示されていた内容**（画像・パラメータ・費用）を記録に残す
- [ ] actor（誰が）・role（どの権限で）・timestamp・判定理由を保存
- [ ] 共有APIトークン1本の環境では「誰が」を証明できないことを**明記**し、承認者名の自由入力なら「自己申告」であるとUI/台帳に明示

### 2. 失効（古い承認を生かさない）
- [ ] **入力の版が変わったら承認は自動失効**（種画像差し替え→ポーズ承認は無効、のような下流連鎖）
- [ ] 承認の取消（revocation）操作と履歴
- [ ] 「最新の承認1件を拾う」実装は差し替え競合に弱い — 承認↔対象版の外部キーで結合

### 3. 状態機械との統合（承認待ちを殺さない）
- [ ] 承認待ちは専用状態（例: `awaiting_approval`）。running/pending に相乗りさせない
- [ ] **タイムアウト reaper・起動時 sweep の対象から承認待ちを除外**（人間は分〜日単位で応答する）
- [ ] 承認→次工程の遷移は CAS（現状態を条件に UPDATE）で二重承認・レース封鎖

### 4. 合格基準の定義（承認のブレ防止）
- [ ] 「OKに見える」ではなく**判定票**（観察項目の列挙。例: 顔・年齢感・髪・輪郭・衣装・指の破綻・過度な美化の有無）
- [ ] 複数承認者が要る判断（法務・肖像）と単独でよい判断を区別
- [ ] 不合格時の差し戻し先（再生成か・素材変更か）を状態遷移に含める
