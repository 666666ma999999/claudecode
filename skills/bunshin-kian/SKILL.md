一言のテーマを受け取り、①現況を先に読み ②本人の4段テンプレで起案文を組み ③本人にしか出せない値だけ質問し返す。

> 型の根拠 = `~/.claude/docs/bunshin-taxonomy-v2.md` §11.5(正確な癖テンプレ)・§11.3(品質基準)・§3(人間専権12スロット)。原文実引用つき。

## 生成手順(必ずこの順・省略禁止)

### STEP 0 — 現況を先に読む(本人の #1 基準 "現運用101件"・省略厳禁)
cwd から project を特定し、**現況を実取得してから**でないと起案文を書かない:
- `tasks/NOW.md`(優先順位) / `plan.md`(設計) / `docs/measures-detail.md`(施策) / `data/raw/` の最新 / `metrics/`
- 未取得なら「現況未確認」と明示し、断定を弱める。**直感・雛形・一般論だけで起案しない**。

### STEP 1 — 人間専権スロットの穴埋め質問(先に聞く・AskUserQuestion 最大4問)
本人の頭からしか出ない値を、分身は**勝手に数値で埋めない**(それらしい嘘閾値を作らない)。テーマに該当するものだけ聞く:
- **相場観・数値閾値**(何をファクトと呼ぶか。例「2週間で10枚以上売買成立ならファクト」)
- **採算ライン**(粗利/真ROAS/許容CPA/予算枠)
- **投資thesis・勝ち筋の仮説**(例「英語版が先に上がる」)
- **会議・組織で決まった事項**(あれば)
答えを起案文に埋める。分からないと言われた項目は「未確定(要決定)」として起案文に残す。

### STEP 2 — 4段テンプレで起案文を組む(本人の型・4/30 初出・全project共通)
出力する起案文は必ずこの構造:
1. **【背景】** なぜ今これか(会議決定 or 現状数値。STEP 0 で読んだ実データを1-2行)
2. **【検討してほしいポイント】** 番号列挙。**答えは出さず論点だけ**置く(本人流)
3. **【成果物】** 形式と置き場所を**絶対パス**で(読み物=vault MD / コード=repo。「開いて」と言われる前に置き場所指定)
4. **【検証】** 「**codex , agent team で考え → 敵対的レビュー → ファクトチェック(一次ソース照合・捏造禁止・出典 file:line)**」を明記
埋め込む品質基準(hard precondition・本人のBEST項目): 現運用を見てから / 出典必須 / 二重記載を作らない / 非エンジニアに平易(BLUF) / 既存資産を先に棚卸し(新手段に飛ばない)。

### STEP 3 — 承認と着手を分離(承認ミニマル)
完成した起案文を提示し、**「このまま投げる / 直す」を一言で聞く**。承認("a"級の一言)を得てから初めて実行委譲へ。分身は計画と着手を1発話に混ぜない。

## 出力形式(この順)
- **BLUF(3行)**: この起案で何を狙うか
- **起案文本体**(コピペ即投入可・本人の文体・4段構造)
- **codex に叩いてほしい論点 3つ**(本人が必ず付ける自己敵対レビュー)
- **保存の提案**: `<project>/prompts/<project>_INBOX.md` の 🔵 欄へ保存するか聞く(offer・勝手に保存しない)

## やってはいけない(負例・本人が最も嫌う)
- 現況を読まずにテンプレだけ埋める(現運用軽視 = 本人の最頻是正)
- 人間専権スロットを勝手に"それらしい数値"で埋める(嘘閾値)
- AIの委譲文体(役割付与の長文prose)をそのまま量産する(本人の癖でなくAIの癖 = モデル崩壊)
- 完了・反映を観測せず自己申告する(surface-compliance)

## v0 の限界(正直に)
- 「ゼロイチのひらめき→起案」の着想そのものは生成しない(記録が薄い)。テーマは人間が差す。
- 「基準をどの強さで要求するか」の較正は未実施(壊れたAI相手のデータのため)。
- 事業ドメインの数値は STEP 1 で必ず人間に聞く(埋められない)。
