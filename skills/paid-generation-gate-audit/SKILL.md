# 有償生成ゲート監査

有償の生成APIを呼ぶ機能（発注UI・バッチ量産・自動リトライ）を実装/レビューするとき、以下を全点検する。1つでも欠けると「気づいたら数十ドル」事故になる（出典: reading-factory Phase C 敵対レビュー 2026-07-14）。

## チェックリスト

### 1. 実行前ゲート（estimate → confirm）
- [ ] 実行前に**見積額を人間に表示**し、明示 confirm なしで課金処理が始まらない
- [ ] confirm token は **hash(全課金パラメータ)** に束縛（モデル・秒数・音声有無・解像度・本数を含む）。安い見積で token を取り実行時に膨らませる攻撃を封鎖
- [ ] token に TTL と**単回消費**（並行 confirm 二重送信でも provider 発注は1回）
- [ ] 見積式と実請求の一致を検算（例: 初期8秒+延長10秒契約なら $0.80 でなく $1.00。表示額に延長・音声・リトライ分を含める）

### 2. 冪等性・復旧（二重課金の主戦場）
- [ ] provider の **operation ID を送信前に永続化**（送信前/送信済み/結果回収済みの3状態を区別）
- [ ] クラッシュ・再起動後は「failed 扱いで再実行」ではなく **operation ID で再照会**して状態復元（再実行=二重課金）
- [ ] 自動リトライは回数上限つき・**失敗分の課金有無を確認**して cost 台帳に二重計上しない

### 3. 上限（多層）
- [ ] 1回あたり上限（本数×単価）
- [ ] **日次コスト上限**（config 定数 + 実行時に当日合計をクエリして超過 429）
- [ ] グローバル同時有償ジョブ数上限
- [ ] 上限判定と INSERT が単一トランザクション（TOCTOU 封鎖）

### 4. 迂回路の封鎖
- [ ] **既存の別エンドポイントが confirm を迂回できないか** rg で全 provider 呼び出し経路を列挙（例: 新設 Phase C に confirm を付けても既存 /api/videos/* が素通しなら無意味）
- [ ] 課金権限は router ごとでなく**共通サービス層**で強制

### 5. 台帳
- [ ] cost_usd は「実際に課金された分」のみ記録・failed/リトライの扱いをコメントで明文化
- [ ] 誰が confirm したか（承認記録 → human-approval-design スキル参照）
